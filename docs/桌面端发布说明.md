# 小说书库桌面端发布说明

## 版本来源

每次发布必须同步更新以下位置，且版本号完全一致：

- 根目录 `package.json`
- `apps/desktop/package.json`
- `apps/desktop/src-tauri/Cargo.toml`
- `apps/desktop/src-tauri/tauri.conf.json`
- `CHANGELOG.md`
- `releases/releases.json`

执行以下命令检查：

```bash
npm run release:validate
```

## 更新签名

当前签名材料保存在本机：

```text
E:\customize\book\.release-local\signing\novel-library.key
E:\customize\book\.release-local\signing\novel-library.key.pub
E:\customize\book\.release-local\signing\novel-library-key-password.xml
```

私钥和密码文件不得提交。密码文件由 Windows 当前账户加密，只能在当前账户下解密。

本机签名材料、手工下载的正式安装包和临时发布资料统一放在工程内已被 Git 忽略的 `E:\customize\book\.release-local`。常规编译产物使用 `apps\desktop\src-tauri\target`。不得将任何开发或发布文件写入用户安装目录 `D:\Software\NovelLibrary`，该目录只允许包含安装器管理的应用文件，确保卸载后可以被完整移除。

GitHub 仓库需要配置 Actions Secrets：

- `TAURI_SIGNING_PRIVATE_KEY`：私钥文件的完整内容。
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`：私钥密码明文。

## 发布流程

1. 所有功能、修复和发布准备从 `develop` 或基于 `develop` 的功能分支提交，禁止直接在 `main` 提交或推送。
2. 更新版本号、`CHANGELOG.md` 和 `releases/releases.json`。
3. 将版本记录的 `published` 设为 `true`，确认 Release 与安装包 URL。
4. 执行 `npm test`、`cargo test`、`npm run desktop:web:build` 和 `npm run release:validate`。
5. 验证通过后将 `develop` 合并到 `main`，并推送合并结果。
6. 创建并推送相同版本的 Tag，例如 `v0.1.0`。
7. `.github/workflows/release-desktop.yml` 构建签名 NSIS 安装包、`.sig`、`latest.json` 和 SHA256 文件，并创建 GitHub Release。
8. 安装上一版本，验证自动检查、下载、覆盖安装、重启和原数据库读取。

首次克隆后执行以下命令启用仓库内 Git Hooks：

```bash
git config core.hooksPath .githooks
```

## 历史版本

应用内历史记录来自 `releases/releases.json`。用户选择旧版本时打开对应 GitHub Release 安装包。数据库只做向上迁移，不做向下迁移，因此降级前必须导出 JSON 备份。
