# 小说书库桌面端发布说明

## 版本来源

`releases/releases.json` 是版本说明、数据兼容信息、安装包地址和校验值的唯一来源。每次发布必须同步更新以下位置，且版本号完全一致：

- 根目录 `package.json`
- `apps/desktop/package.json`
- `apps/desktop/src-tauri/Cargo.toml`
- `apps/desktop/src-tauri/tauri.conf.json`
- `CHANGELOG.md`
- `releases/releases.json`

每条版本记录必须包含 `published`、`minimumSupportedVersion`、`requiresBackup`、固定 Tag 的 Release/安装包 URL、SHA256、更新章节和升级说明。版本记录只允许在首位新增；正式发布后禁止改写或删除历史记录。

发布前执行：

```bash
npm test
cargo test --manifest-path apps/desktop/src-tauri/Cargo.toml
npm run desktop:web:build
npm run release:validate
```

CI 还会使用上一提交作为历史基线：

```bash
npm run release:validate -- --tag=v0.5.0 --baseline-ref=v0.5.0^1
```

## 更新签名

当前签名材料保存在本机：

```text
E:\customize\book\.release-local\signing\novel-library.key
E:\customize\book\.release-local\signing\novel-library.key.pub
E:\customize\book\.release-local\signing\novel-library-key-password.xml
```

私钥和密码文件不得提交。密码文件由 Windows 当前账户加密，只能在当前账户下解密。GitHub 仓库需要配置：

- `TAURI_SIGNING_PRIVATE_KEY`：私钥文件完整内容。
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`：私钥密码明文。

发布权限与签名密钥以 GitHub 仓库的 `main` 发布链路为准。工作分支和本地环境不要求持有 GitHub Release 密钥，也不要求本机 `gh` 登录；本地只负责完成版本准备、测试和推送工作分支，合并到 `main` 并推送版本 Tag 后，由 GitHub Actions 从仓库 Secrets 读取签名材料完成构建和正式发布。禁止因为本机未登录或没有密钥而改为从工作分支直接发布，也禁止绕过 `main` 直接推送发布 Tag。

本机签名材料、手工下载的正式安装包和临时发布资料统一放在已忽略的 `.release-local`。常规编译产物使用 `apps/desktop/src-tauri/target`。不得将开发或发布文件写入用户安装目录；安装目录只包含安装器管理的应用文件。

## 发布资产

Windows x86_64 NSIS 正式 Release 至少包含：

```text
NovelLibrary_<version>_x64-setup.exe
NovelLibrary_<version>_x64-setup.exe.sig
NovelLibrary_<version>_x64-setup.exe.sha256
latest-windows-x86_64-nsis.json
latest.json
legacy-latest.json
SHA256SUMS.txt
```

- `latest-windows-x86_64-nsis.json` 是新客户端使用的固定版本目标 Manifest。
- `latest.json` 是旧客户端通过 `/releases/latest/download/` 使用的兼容入口。
- `legacy-latest.json` 保存发布开始前的上一版 `latest.json`，用于审计和故障恢复。
- `.sig` 由 Tauri updater 验证；`.sha256` 和 `SHA256SUMS.txt` 供公开校验。

## 发布流程

1. 所有功能、修复和发布准备在工作分支完成。工作分支可以是 `develop`、`feature/*`、`fix/*` 或其他临时分支；禁止直接在 `main` 提交或推送。
2. 在版本目录首位新增记录，更新所有版本文件和 `CHANGELOG.md`，提前用最终安装包填写 SHA256。
3. 完成本地测试和 `npm run release:validate`。
4. 将本次发布涉及的工作分支（可以是一个或多个）审核合并到本地 `main`，确认 `main` 包含完整发布内容；发布来源始终是 `main`，不能从工作分支创建 Release Tag。
5. 将 `main` 推送到 `origin/main`，再从 `main` 创建并推送同版本 Tag，例如 `v0.5.0`。
6. `prepare-release` 校验 Tag 和历史记录，创建 Draft Release，并把上一正式版 `latest.json` 保存为 `legacy-latest.json`。
7. `build-windows` 构建签名 NSIS 安装包，生成目标 Manifest、兼容 Manifest 和单包 SHA256，上传暂存资产。
8. `upload-checksums` 从暂存 Release 重新下载安装包并生成 `SHA256SUMS.txt`。
9. `finalize-release` 先把 Release 公开为 Prerelease，用固定 Tag 地址验证安装包、签名、SHA256 和目标 Manifest。
10. 固定地址验证通过后才上传最终 `latest.json`，晋升为正式 Latest，并验证 `/releases/latest/download/latest.json`。
11. 最终验证失败时 Workflow 自动将 Release 降回 Prerelease，旧客户端继续读取上一正式版本。

### 分支合并检查

发布前至少确认以下事项：

```bash
git switch main
git pull --ff-only origin main
git merge --no-ff <工作分支>
npm test
cargo test --manifest-path apps/desktop/src-tauri/Cargo.toml
npm run desktop:web:build
npm run release:validate
git push origin main
git tag v<version>
git push origin v<version>
```

如果本次发布包含多个功能或修复分支，逐个审核合并到 `main` 后再统一验证和打 Tag。任何 Tag 如果没有被 `origin/main` 包含，Release Workflow 会拒绝发布。发布过程中遇到本地 GitHub CLI 未登录时，不应要求用户提供密钥或改走工作分支发布；应先完成 `main` 合并，再由 Actions 使用仓库 Secrets 发布。

已正式发布的版本禁止通过重跑修改。内容有误时发布新的修复版本。Draft 或失败后留下的 Prerelease 可以幂等重跑，资产会在重新校验后覆盖。

## 手工生成与验证

调试发布资产时可执行：

```bash
npm run release:manifest -- \
  --version=0.5.0 \
  --manifest=releases/releases.json \
  --installer=NovelLibrary_0.5.0_x64-setup.exe \
  --signature=NovelLibrary_0.5.0_x64-setup.exe.sig \
  --output=latest-windows-x86_64-nsis.json \
  --latest-output=latest.json \
  --checksum-output=NovelLibrary_0.5.0_x64-setup.exe.sha256
```

公开后验证：

```bash
npm run release:verify -- \
  --repository=kengqin/book \
  --tag=v0.5.0 \
  --manifest=releases/releases.json \
  --expected-state=stable \
  --verify-latest=true
```

验证器会读取 GitHub Release API 和公开下载地址，核对 Release 状态、必要资产、Tag/Manifest 版本、官方安装包 URL、签名文本、安装包 SHA256、单包校验文件以及兼容平台键。

## 发布后回归

- 使用上一正式版检查新版本，确认不会在 Manifest 同步期间错误显示“当前已是最新版本”。
- 验证下载进度、取消、重试、跨页面全局状态、安装失败重试和安装后重启。
- 验证自定义安装目录、自定义数据目录、自定义数据库路径、TXT、EPUB、阅读进度和笔记均保持不变。
- 验证历史版本入口仍可打开；降级前必须导出完整备份，数据库不做反向迁移。

首次克隆后执行以下命令启用仓库内 Git Hooks：

```bash
git config core.hooksPath .githooks
```
